#00简介

龙虾训练是 OpenClaw 把 AI 智能体打造成靠谱运营分析师的方法论。每一只"龙虾"都会被赋予明确的职责、数据边界、运行节奏与行为人格。它不是聊天机器人,而是数字员工——按时醒来,核对账目,生成看板,推送早报,然后回去休眠。

从零到上线要走过 六个里程碑:环境就绪 → 工具配置 → 聊天通道 → 闭环任务 → 系统对接 → 分析大脑。六步走完,即可投产;漏掉任何一步,日后就要天天救火。

本手册适用对象

开发者与分析师:负责龙虾的日常训练。技术负责人:评审系统集成方案。业务方:想了解一只训练有素的生产龙虾到底能做什么。

#01启动前清单

动手安装之前,请先核对下表。绝大多数生产事故的根源,都出在跳过了这一步。

项目最低要求推荐配置
操作系统macOS 12+、Ubuntu 20.04+、Windows 11 + WSL2Ubuntu 22.04 LTS
CPU / 内存 / 磁盘4 核 · 8 GB · 20 GB SSD8 核 · 16 GB · 50 GB SSD
网络可对外 HTTPS 访问 OpenClaw、飞书/Telegram 及业务系统同上,另配静态出口 IP
账号OpenClaw 工作区、飞书管理员或 Telegram bot token,以及财务/CRM/ERP 的只读凭据为龙虾新建独立服务账号,并开启审计日志
最佳实践 —— 先用只读账号

所有系统一律从只读凭据开始,等端到端行为验证通过之后,再逐个开放写权限。这一条规矩能挡掉 80% 的尴尬生产事故。

常见陷阱 —— 共享管理员账号

绝对不要把任何同事的个人管理员密码直接给龙虾。务必使用独立的服务账号,否则每一条审计日志看起来都像你 CFO 凌晨三点亲自跑 SQL。

#02下载、安装与配置

龙虾技术栈自下而上分四层,请按顺序安装,后一层依赖前一层。

2.1 — 运行时

先把运行时装在一台主机上,它就是你的"控制面"——负责调度、监控和派遣龙虾。

macOS · Linux
curl -fsSL https://get.openclaw.dev | sh
openclaw doctor   # 12 项健康检查
Windows · PowerShell
iwr https://get.openclaw.dev/install.ps1 -useb | iex
openclaw doctor

openclaw doctor 是本手册里最关键的一条命令。任何一行返回红色,就先解决那一行。它会依次检查 Node、Python、磁盘、网络、时钟偏差,以及对 OpenClaw 云端的连通性冒烟测试。

2.2 — 大模型配置

龙虾的"大脑"是大模型。选定供应商后,务必在配置中锁死模型版本,这样多次运行的行为才稳定可复现。

~/.openclaw/config.yaml
llm:
  provider: openai
  model: gpt-4o-2024-08-06
  temperature: 0.2
  max_tokens: 4096
最佳实践 —— 锁死版本

版本号、模型名、温度参数,全部都要锁。训练时使用 gpt-4o-2024-08-06 的龙虾,一旦该模型 id 退役,行为立刻漂移。要锁快照,不要锁别名。

2.3 — 连接器

按需安装连接器,缺什么装什么。凭据统一存进操作系统的钥匙串——绝不写入明文 YAML。

shell
openclaw connector add finance --type=kingdee
openclaw connector add crm     --type=salesforce
openclaw connector add erp     --type=sap

2.4 — 调度器

龙虾不知疲倦,但贵在守时。调度器是一个轻量的 cron 风格执行器,按你指定的时间把龙虾叫醒。

shell
openclaw scheduler enable
openclaw scheduler add daily-morning "0 8 * * 1-5" report.daily
openclaw scheduler add weekly-board  "0 9 * * 1"   board.refresh
openclaw scheduler add hourly-watch  "0 * * * *"   alerts.sweep
常见陷阱 —— 本地时区 vs. 业务时区

调度器按主机的本地时区执行。如果 CFO 在新加坡、服务器在法兰克福,请显式设置时区:openclaw scheduler tz Asia/Singapore。否则龙虾会用昨天的数据向新加坡问好。

#03飞书与 Telegram integration

没法和人对话的龙虾,只是个脚本。选一个聊天通道把它接出来——生产用一个,另一个作备份。

3.1 — 飞书

  1. 在飞书管理后台创建一个自建应用,名字取作 OpenClaw 龙虾(或自取)。
  2. 勾选权限范围 im:message、im:message.group_at_msg、im:message.p2p_msg、im:message:send_as_bot。
  3. 开启机器人能力,并复制 App ID、App Secret、Encrypt Key 与 Verification Token。
  4. 在 OpenClaw 中注册机器人:
shell
openclaw chat add feishu \
  --app-id=cli_xxxxxxxxxxxxxxxx \
  --app-secret=xxxxxxxxxxxxxxxxxxxxxxxx \
  --encrypt-key=xxxxxxxxxxxxxxxxxxxxxxxx \
  --verification-token=xxxxxxxxxxxxxxxx
  1. 订阅事件 im.message.receive_v1,请求 URL 填 https://<your-host>/webhook/feishu。
  2. 把机器人加进任意群聊,发送 /lobster hello,收到回复即表示端到端鉴权已通。

3.2 — Telegram

  1. 在 Telegram 里找到 @BotFather,执行 /newbot,取名 OpenClaw 龙虾。
  2. 复制 HTTP API token。
  3. 注册并设置 webhook:
shell
openclaw chat add telegram --token=123456:ABC-DEF

curl "https://api.telegram.org/bot<TOKEN>/setWebhook?url=https://<your-host>/webhook/telegram"
常见陷阱 —— Webhook 超时

Telegram 会在 60 秒后取消请求。耗时的分析任务(预测、深度诊断)必须异步回复——先回执,后端跑完再回结果。只要在命令里加 async: true,运行时会自动处理。

最佳实践 —— 一只龙虾一个机器人

不要在两只龙虾之间共享同一个 bot token。聊天记录会串,排查"幽灵消息"会浪费一整个下午。一只龙虾,一个机器人,一条独立 webhook,一份独立日志。

3.3 — 常用命令

命令说明
/status健康状态、上次运行、下次计划
/dash revenue推送收入看板
/forecast --horizon=90跑一次 90 天销售预测
/ask <question>自然语言查询
/alert set budget-variance=5%调整告警阈值
/run daily手动触发一次日常任务

#04闭环龙虾任务

"任务"是一段端到端的固定流程,可手动触发,也可定时调度。一只生产级龙虾通常持有三套任务:日常、每周、临时。其中日常任务是它的"心跳"。

日常任务的七个阶段

每只生产级龙虾的日常运行,都按以下七个阶段顺序执行:

1
唤醒 & 阅读
2
Extract
3
Clean
4
Aggregate
5
Analyze
6
Diagnose
7
Report

夜间唤醒并阅读告警 → 从财务/CRM/ERP 抽取增量数据 → 清洗对账 → 聚合成日粒度表(每天每客户每商品一行)→ 刷新 KPI、同环比、偏差、预测 → 对所有波动超 5% 的指标进行诊断 → 推送早报卡片。这七个阶段缺一不可,这正是龙虾训练的核心要义。

闭环架构
最佳实践 —— 把每个阶段当作一个单元测试

每个阶段都要做到幂等且可独立重跑。早上 8:03 出了岔子,你应该可以只重跑第 4 阶段,不必重新从财务系统拉数。运行时支持 openclaw task run daily --from-phase=4。

#05Connecting 财务, CRM & ERP

这三套系统是龙虾干活的"原料来源"。每套系统对应一个独立连接器,各自配备最小权限。

系统读取回写
财务
Kingdee · Yonyou · NetSuite		GL, AR/AP, cost centers, budget vs. actuals异常标注(可选)
CRM
Salesforce · HubSpot		客户、商机、漏斗、输赢单线索打分、下一步动作建议
ERP
SAP · Oracle · Dynamics		销售订单、库存、出货、BOM预测调整
shell
openclaw connect finance --type=kingdee   --base-url=https://kd.example.com   --account=lobster-svc
openclaw connect crm     --type=salesforce --base-url=https://sf.example.com   --account=lobster-svc
openclaw connect erp     --type=sap        --base-url=https://sap.example.com  --account=lobster-svc

每一套都要跑一次冒烟测试,全绿代表:鉴权通过、表结构已识别、增量同步字段定位成功,且最近一条记录距今 < 24 小时。

shell
openclaw connect test finance
openclaw connect test crm
openclaw connect test erp
常见陷阱 —— 表结构漂移

业务系统改字段名是常态。周一还跑得好好的龙虾,周二如果 ERP 团队把 net_amount 改成了 net_amt,就会当场罢工。运行时会监测这种漂移,并自动暂停流水线;你必须用 openclaw connect approve finance 显式确认新映射才会恢复。

#06日常数据流水线

连接器就绪后,流水线自动接管。每一行数据都会打上 _lobster_run_id 标签,任何数字都能反查到具体是哪一次运行产生的。

shell
openclaw pipeline init \
  --source=finance,incremental=posted_at,dimensions=account,cost_center \
  --source=crm,incremental=updated_at \
  --source=erp,incremental=changed_at \
  --sink=duckdb   # 也可: snowflake | bigquery

一次干净跑下来,每天会产出:fact_revenue_daily、fact_cost_daily、fact_orders_daily,以及 dim_product、dim_customer、dim_date。

最佳实践 —— 一个粒度,一张表

不要在同一张事实表里混日粒度与月粒度。汇总指标应当即时计算而非预先存储。数据湖是"月度汇总"唯一存在的地方,所有看板都要从日粒度推算。

#07收入、成本与利润看板

龙虾出厂自带一套默认看板库。每个看板都是单页交互视图,流水线每次跑完都会自动刷新。

7.1 — 四个核心 KPI

所有看板都覆盖同一个四件套:收入(已确认,扣减退货)、成本(全成本口径)、利润(毛利 / 贡献毛利 / 净利)和偏差(实际 vs. 预算,默认 5% 阈值)。一个 KPI 若没有配套的比较项和阈值,就不算真正的 KPI。

收入
¥48.2M
▲ 环比 +6.1%
成本
¥36.9M
▲ 环比 +3.2%
利润
¥11.3M
▲ 环比 +18.4%
利润率
23.4%
▲ 提升 0.9 个百分点

7.2 — 四种对比

对比方式回答什么问题
环比(MoM)本月比上月强吗?
同比(YoY)相对去年同期有增长吗?
计划 vs. 实际在预算内吗?
预测 vs. 实际预测越来越准了吗?

7.3 — 可视化规范

一张图只回答一个问题。龙虾默认模板就是这样:趋势用折线图配 30 日滚动均线;构成用 100% 堆叠柱或树图;分布用箱线图(绝不用饼图);对比用排序柱状图;KPI用大数字 + 迷你折线 + 涨跌幅标签。

常见陷阱 —— 饼图

7 块的饼图没法看,改用排序柱状图。龙虾默认模板本就禁止饼图——生产看板里如果出现了饼图,一定是有人手贱覆盖了模板,赶紧改回去。

最佳实践 —— 加注释,别堆装饰

每张图都要回答一个问题,并配一行图注。比如"收入环比增长 12%,主要来自 A 产品线。"如果需要写一大段才能讲清楚,说明图本身就该重做。

#08预算跟踪与偏差告警

龙虾的偏差引擎会逐格监控预算中的(科目,期间)单元格。默认阈值为 5%,超出后按严重度分级告警。

~/.openclaw/alerts.yaml
alerts:
  - name: budget-variance
    metric: cost_vs_budget
    threshold: 0.05          # 5%
    direction: both          # 正向或负向偏差
    severity:
      info:     [0.03, 0.05)
      warn:     [0.05, 0.10)
      critical: [0.10, inf)
    route:
      info:     feishu:daily-report
      warn:     feishu:finance-team
      critical: feishu:cfo + telegram:cfo

When the daily pipeline runs, the variance check is automatic. If cost drifts 6% over budget in the Shanghai warehouse, the lobster posts an info card to the daily report, DMs the finance team on Feishu, pages the CFO on Telegram and Feishu, and opens a diagnosis thread with the three most likely root causes.

最佳实践 —— 告警必须配假设

一条孤零零的"偏差 6.2%"是噪音。一条"偏差 6.2%,首要假设:供应商 X 运费上涨 14%"才是可操作的。告警内容里务必带上最可能的假设。

#09销售预测

龙虾出厂自带三套预测模型,以及它们的融合版本。每种模型各有所长,在滚动回测中,融合模型比任何单一模型都更准。

模型适用场景输入
趋势 + 季节性
Holt-Winters		 稳定、长生命周期的老品 2 年以上日度历史销量
Prophet 节假日/大促效应明显的商品 历史销量 + 节假日日历 + 大促日历
机器学习融合
LightGBM		 新品上市、波动较大的 SKU 历史 + 价格 + 宏观信号 + 网站流量
shell
openclaw forecast run \
  --horizon=90d \
  --granularity=daily \
  --models=holt-winters,prophet,ensemble \
  --signals=web_traffic,fx_rate,commodity_index

每条预测都会经过滚动起点回测验证。只有当滚动 MAPE 低于该商品历史基线时,预测才会被发布;否则自动回退到更简单的模型,并打上"待人工复核"的标记。

最佳实践 —— 必须引入市场信号

只用内部历史的预测,根本扛不住宏观冲击。龙虾默认配置会自动引入汇率、大宗商品指数,以及(可选的)网站流量作为外生变量——在波动 SKU 上,MAPE 通常能改善 8–15%。

常见陷阱 —— 别轻信第一次预测

回测、查残差、再发布。第一次跑出来的预测几乎从来都不是最终版。如果残差暴露出漏掉的日历效应,补上;如果呈现周期切换特征,就缩短训练窗口重训。

#10利润诊断与智能报告

只会吐数字的龙虾是计算器,会给出结论的龙虾才是分析师。差异全在诊断引擎。

诊断的四道工序

每发生一次 KPI 波动,龙虾都会跑完这四道工序:

  1. 拆解——把波动拆成量、价、结构、成本四个分量。
  2. 切片定位——找出是哪个产品、地区、渠道或客户推动了这次波动。
  3. 生成假设——基于数据提出 2–4 个合理的根因。
  4. 证据打分——按各假设与数据的吻合度排序。

诊断的输出是一张报告卡,不是看板。下面是一份真实样例:

每日报告 · 2026-06-29

净利润环比增长 8.4%,跑赢预算 1.2 个百分点。

变化概要。收入环比 +6.1%(量 +3.0%,价 +2.1%,结构 +1.0%);成本 +3.2%(其中运费 +14%,是最大单点);利润率 23.4%,提升 0.9 个百分点。

原因初判。A 产品线借力一场成功的大促增长 18%;B 产品线因新品上市推迟下滑 4%。

行动建议
  1. 重新谈判上海线运费,当前价格比区域中位数高 22%。
  2. 把 A 产品线的大促玩法复制到 C 产品线(客群重合度较高)。
  3. 保留 B 产品线的上市预算,Q3 走势会自然恢复。

龙虾会先起草好报告,以卡片形式发到飞书/Telegram;经人工审阅或微调后,再发往更广的接收方。

最佳实践 —— 报告必须以行动建议收尾

没有建议的报告,只是状态播报;有建议的报告,才能推动决策。龙虾已被训练成"绝不以一个数字结尾"——永远以行动建议结尾。

#11自然语言查询

数据湖就绪之后,公司里任何人都可以用自然语言向龙虾提问,龙虾会给出精准、可溯源的答复。

示例

你: 上个月哪个产品线利润增长最高?

龙虾: 是 A 产品线。净利润环比增长 22%(¥480 万 → ¥590 万),主要受 Q2 大促推动,销量提升 18%。其中贡献最大的是 SKU A-12,占增量的 32%。

背后的工作流

  1. 运行时识别出这是一个查询意图(NLQ)。
  2. 查询规划器把自然语言转成结构化计划:{metric: profit, dimension: product_line, time: last_month, comparison: mom}。
  3. 该计划被翻译成 SQL,在数据湖上执行。
  4. 回答生成器输出一段 2–4 句的自然语言答复,并附数据出处。

对于重复问题,龙虾会缓存查询计划。同一道题 100 个人在上午问出来,实际开销只相当于若干次执行,远低于 100 次全新查询。

常见查询类型

"本季度收入前 5 名的客户"——排名类。 "3 月成本为什么飙升?"——诊断类。 "预测 Q4 销售额"——预测类。 "Acme 公司由谁负责?"——CRM 查询类。 "5 月我们达成计划了吗?"——偏差类。 运行时会自动识别查询意图,并把计划以 SQL 形式执行。

常见陷阱 —— 时间窗口的歧义

"上个月"在月初和月末指代的时间段并不一样。龙虾的默认是最近一个完整自然月。如果你想要别的口径,务必明确说出:"过去 30 天"、"2026 年 5 月"、"财年 5 月"。

最佳实践 —— 沉淀查询库

把最常被问到的 20 个问题存为命名查询。前 20 个问题,基本上就是 CFO 每周一早上必问的那批。命名后,响应时间能压到亚秒级,而且有稳定 URL 可以直接分享。

#12总结与速查

六个里程碑清单

  • M1 — 环境。操作系统、内存、磁盘、网络、账号、审计日志。
  • M2 — 工具。OpenClaw 运行时、大模型锁版本、连接器装齐、调度器启用。
  • M3 — 聊天通道。飞书或 Telegram 机器人建好,webhook 验证通过。
  • M4 — 任务。日常闭环任务定义完毕,7 个阶段全部幂等。
  • M5 — 系统对接。财务、CRM、ERP 全部连通,冒烟测试全绿。
  • M6 — 分析能力。看板上线、告警调参、预测验证、查询库就位。

四个核心 KPI

收入、成本、利润、偏差——铁律:一个 KPI 若没配比较项和阈值,就不算真正的 KPI。

唯一铁律

报告永远以行动建议收尾。数字只能描述过去,建议才能改变未来。

#A上线前检查清单

准备投产的那个早上,逐条过一遍这十项。十条全绿,放心上线。

  • openclaw doctor 所有项均绿。
  • openclaw pipeline run daily 在 15 分钟内成功完成。
  • openclaw status 显示下一次计划运行时间在未来。
  • 用合成的超预算数据触发,告警能正常发出。
  • 每条活跃产品线的滚动 MAPE 都低于基线。
  • 一份样例诊断报告已被人工审阅并通过。
  • 一道样例自然语言查询能正确返回结果。
  • 飞书/Telegram 机器人对 /status、/dash、/ask 均能正确响应。
  • 审计日志已成功汇入 SIEM。
  • 值班同学已知晓这只龙虾的存在,并掌握 openclaw pause 暂停方法。
