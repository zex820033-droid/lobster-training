1下载、安装与工作区初始化

OpenClaw 是「生命容器」：Agent 不是 prompt，是独立运行单元（参见八卷·卷三·模块1）。先把容器装好，再谈训练。

1.1 主机要求（来自 SHARED/TOOLS.md）

项目要求

Shellzsh（默认即可）

1.2 安装 OpenClaw 与初始化工作区

安装 CLI

通过官方安装脚本拉取 OpenClaw 运行时。

3接入飞书 / Telegram：让龙虾会「喊人」

闭环任务的命脉是「异常能触达人」。经营分析师 a02 的 KPI 之一就是 异常预警及时率 ≥ 90%（异常→预警 ≤ 4h）。下面把两条通知通道接上。

飞书（Feishu）接入

适合企业内、正式汇报与群聊协同（12 虾协同频道）。

  1. 在飞书开放平台创建「自建应用」，获取 App ID / App Secret
  2. 为目标群创建 自定义机器人，拿到 Webhook URL
  3. 申请权限：im:message、im:chat
  4. 写入 SHARED/TOOLS.md 的「网络与通知」段
# 配置到全局工具
$ openclaw channel add feishu \
    --app-id $FEISHU_APP_ID \
    --app-secret $FEISHU_APP_SECRET \
    --webhook $FEISHU_WEBHOOK

Telegram 接入

适合移动端即时推送、跨境团队与个人告警。

  1. 找 @BotFather 创建 Bot，拿到 Bot Token
  2. 把 Bot 拉入群组，获取 chat_id
  3. 写入 SHARED/TOOLS.md
$ openclaw channel add telegram \
    --token $TG_BOT_TOKEN \
    --chat-id $TG_CHAT_ID

3.1 路由策略：谁的消息发哪里

在 a02 的 HEARTBEAT.md 里定义熔断上报条件与通道映射，遵循八卷·卷二《HEARTBEAT 文档》：

事件级别触发条件主通道备用通道
P0 紧急预算偏差 > 5% 且金额 > 阈值 / 利润转负飞书群 @CEOTelegram
P1 警告单指标环比波动 > 20%飞书群无
P2 提示趋势信号（连续 2 周同向）日报摘要无
实测要点：飞书 Webhook 对单条消息有长度限制，长报告先写文件再发链接；Telegram Bot 在群内默认不读消息，需 /setprivacy 关闭才能接收 @ 提及。

4.1 用 Skill 封装连接器（Work-to-Skill 机制）

遵循《蓝血军团》「完成任务→评估→封装」的 Work-to-Skill 机制，把每个系统的连接逻辑沉淀成独立 Skill，集中存放：

Skill数据源用途优先级
connector-finance财务系统 API / 导出表拉取三张表、预算台账P0 必须
connector-crmCRM API订单、客户、退货、复购P0 必须
connector-erpERP API / 库存快照采购成本、库存周转、履约时效P1 推荐
connector-market-signal外部行情/汇率/指数销售预测的市场信号输入P2 可选
# 在集中 Skill 库新建连接器（不写进虾目录）
$ cd workspace/skills
$ openclaw skill new connector-finance --source finance-api
$ openclaw skill grant connector-finance --to i01,a02
安全边界：API Key / 数据库密码不进任何 Markdown，统一放 .openclaw/secrets，由运行时注入。遵循八卷·卷六《自动化治理边界》：没有边界就没有安全。

5每日数据聚合与清洗

这一步直接来自 a02 的「周报生成 SOP · Step 1」（见其 AGENTS.md）。我们把它改造成每日自动触发的数据管线。

5.1 五步清洗流程（对标 a02 工作流）

1

采集汇总

向 d01/e01/f01/g01/i01 发采集请求；六源到位才继续，缺漏则标记并催促。

2

完整性校验

逐源校验行数、主键唯一性、必填项。

3

口径标准化

统一币种、时区（Asia/Shanghai）、单位、会计期间。这是最容易出错的环节。

4

异常初筛

与上一周期对比，波动 > 20% 的指标标记「待分析」。

5

落湖归档

产出标准化数据汇总表 + 异常指标初筛清单，写入 data/ 并归档。

5.2 用 Heartbeat 每日触发

在 a02 的 HEARTBEAT.md 写入每日 08:00 触发的 cron（八卷·卷三·模块6）：

# a02 / HEARTBEAT.md
- cron: "0 8 * * *"
  job: data-aggregation
  on_fail: notify(P1, feishu)   # 管线失败即报警
  circuit_breaker: 连续 2 天失败则熔断并上报 i01
最佳实践：把清洗结果做幂等：同一日重复跑结果一致，避免补数时把历史改坏。
常见坑：跨币种汇总时，用结算日汇率还是订单日汇率，必须固定一种并写进口径文档，否则同比会失真。

同比（YoY）怎么看

剔除季节性的「真增长」。与去年同期比，看 结构性变化，例如品类升级、客户结构迁移。

环比（MoM）怎么看

看「最近做对了什么 / 做错了什么」。连续 2 周同向 = 趋势信号；连续 3 周 = 趋势确认（a02 趋势分级）。

方法学溯源：看板指标选择遵循 skills 库中 kpi-dashboard-design 的 6 标准：可衡量 / 可影响 / 有基准 / 够敏感 / 不被游戏 / 够简单。

7预算偏差追踪与 > 5% 预警

a02 的核心 KPI 之一：预算偏差率 ≤ 15%。本手册进一步把「单科目超 5%」设为自动告警阈值，把事后分析变成事前拦截。

08:14

7.2 偏差判定逻辑

deviation = (实际 - 预算) / 预算
if abs(deviation) > 0.05:
    notify("P0/P1", feishu)          # 主通道
    open_remedy_ticket("预算偏差整改单") # 卷六·模块4 整改单
治理闭环：超 5% 不仅要告警，还要按八卷·卷六开「整改单」；没有纠偏就没有进化。整改单要写明：偏差科目、金额、根因假设、责任人、整改时限。

8销售预测：历史 · 季节 · 市场信号

a02 在华为建立的「区域量价分解模型」= 出货量 × ASP × 区域系数 × 渠道系数。我们把它升级为三因子预测。

历史趋势

移动平均 + 同比基数，锚定基线。

季节因子

旺季系数、促销日、节假日修正。

市场信号

汇率、品类指数、竞品动态（b01 情报）。

8.1 预测公式（简化版）

# 预测下月销售额
forecast = baseline(历史3月加权) × season_factor(月份)
        × market_signal(汇率/指数) × confidence
最佳实践：永远给预测配 置信区间（如 ±8%）和 回测准确率。光给一个数字的分析师不是分析师，是算命先生。
需进一步探索：市场信号权重目前为经验值，建议积累 2–3 个季度数据后用回归重新校准。

9盈利模型自动诊断

不是堆数字，是回答「接下来该做什么」。系统自动跑一遍盈利归因，定位「谁在赚钱、谁在流血」。

维度诊断问题输出
品类哪个品类毛利最高 / 最低？毛利瀑布图 + 排名
渠道哪个渠道 ROI 为正？获客成本 vs LTV渠道盈亏表
客户新客 vs 老客利润贡献？复购率走向？队列分析（cohort）
成本成本结构里哪一项异常上升？成本归因树
现金回款是否健康？渠道库存是否压货？经营性现金流预警

能力支撑来自 skills 库：business-analyst · financial-analyst · data-storytelling · thinking-nate-silver。

关键原则（来自 a02 的 SOUL）：「营收不等于回款，回款不等于健康。」诊断必须穿透到现金流，而不是停在利润表。

10分析报告：结论与建议，而非数字

养虾人（CEO 凌霄）的汇报偏好：金字塔式：结论 → 理由 → 背景；给选项而非给问题；A/B 方案 + 推荐。报告必须照此排版。

10.1 标准报告骨架

# 月度经营分析报告 · 2026-06
## ① 结论（一句话）
本月营收达标但利润承压，物流成本是主要拖累，建议 Q3 启动物流商重新议价。

## ② 核心指标（看板摘要，≤6 项）
营收 ¥3,842k（MoM +12.4%）｜毛利 ¥932k（MoM +2.1%）｜毛利率 24.3%（▼2.5pp）

## ③ 异常归因（谁在拖累 / 谁在拉动）
- 🔴 物流成本超预算 7.3% → 根因：海运涨价 + 退货反件
- 🟢 品类 A 毛利率提升 3pp → 拉动：产品结构升级

## ④ 预测与建议（A/B + 推荐）
- 方案 A：维持现状，预计 7 月毛利率 23.5%
- 方案 B：切换物流商 + 优化退货流程，预计 25.8% ✅ 推荐
- 决策时限：7 月 5 日前

## ⑤ 附录（明细数据 / 三证验真链接）
三证验真（卷六·模块3）：每个关键结论附 截图 / 日志 / 数据 三证链接。没有证据就不是完成。
常见坑：报告不要用「可能、大概、或许」。拿不准的标为「需进一步探索」，并给探索路径。

11自然语言问答

系统必须能直接回答这类问题：

🙋 CEO 凌霄
上个月哪条产品线利润增长最高？
🦞 a02 经营分析师
品类 A（智能音箱），毛利环比 +34.2%，是上月最高增长。

理由：① 产品结构升级，高配款占比从 28% → 41%；② 促销集中在毛利更高的 SKU；③ 物流单价因体量议价下降 4%。

建议：7 月可把广告预算向品类 A 倾斜 15%，预计再拉动毛利 +2pp。
📎 数据出处：CRM 订单 + 财务三张表（三证已附）

11.1 问答如何实现

  • 自然语言 → 结构化查询（基于已清洗数据湖的 schema）
  • 结果 → 经 a02 的「金字塔汇报」模板包装成结论 + 理由 + 建议
  • 每个回答必须带 数据出处（三证），拒绝无来源数字
  • 拿不准时触发 让位协议：转交 i01 财务核验再回复
让位协议（卷七·模块4）：不抢答比能回答更重要。涉及财务口径的问题，a02 让位给 i01。
